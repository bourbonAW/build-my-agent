# Agent Harness 设计哲学

> 原文来源：[Harness Design for Long-Running Application Development](https://www.anthropic.com/engineering/harness-design-long-running-apps)
> 作者：Prithvi Rajasekaran（Anthropic Labs）

---

## 一、原文核心内容

### 背景与问题

该文章探索如何通过多智能体架构（Generator-Evaluator，灵感来自 GAN）提升 Claude 在两个难题上的表现：**前端设计质量**和**自主全栈开发**。

长时运行 agent 存在两个根本性失败模式：

**1. 上下文管理失效（Context Management Issue）**

模型在上下文窗口趋于饱和时会丧失连贯性。Claude Sonnet 4.5 出现了具体可观察的行为——"context anxiety"：在窗口即将填满时过早收尾，不是因为任务完成，而是因为模型感受到上下文压力。实验表明，**重置上下文（context reset）优于压缩上下文（compaction）**：清空并重启提供干净的起点，而原地摘要会引入失真和遗漏。

**2. 自我评估偏差（Self-Evaluation Bias）**

"当 agent 被要求评估自己的输出时，它会自信地赞扬平庸的结果。"——在主观性强的任务上尤为突出。将生成与评估分离，建立外部反馈回路，才能驱动真正的改进。

---

### 前端设计：将主观性转化为可测量标准

前端设计的难点在于质量主观性强。解决方案是将其分解为可量化维度：

| 维度 | 含义 |
|------|------|
| **Design Quality（设计质量）** | 颜色、字体、布局的视觉一致性 |
| **Originality（独创性）** | 刻意的创意选择，而非模板套用 |
| **Craft（工艺）** | 排版层级、间距、对比度等技术执行 |
| **Functionality（功能性）** | 独立于美学的可用性 |

评估 agent 使用 Playwright MCP 与真实页面交互，截图并研究实现后打分。经 5-15 次迭代，设计显著演进——例如一个博物馆网站从标准布局转变为创新的 3D CSS 渲染空间。

---

### 全栈开发：三智能体架构

```
Planner → Generator → Evaluator
```

**Planner Agent（规划者）**

- 将 1-4 句话的简短需求扩展为完整产品规格
- **强调范围（scope）而非细粒度实现细节**——规划的职责是划定边界，而不是预先决定每一步怎么做
- 自然地整合 AI 功能

**Generator Agent（实现者）**

- 使用 React、Vite、FastAPI、SQLite/PostgreSQL 实现功能
- 按 sprint 逐步推进，交付前先自我评估

**Evaluator Agent（评估者）**

- 使用 Playwright 像真实用户一样测试功能，检查数据库状态和 API 端点
- 与 Generator 协商"sprint contracts"——在编码前约定好成功标准，防止范围蔓延

**File-Based Handoffs（基于文件的交接）**

三个 agent 之间通过结构化文件传递工件（artifacts），而非依赖共享内存或对话上下文。这是多智能体分工得以运作的具体机制——当 agent 角色分离后，接力棒必须以可持久、可审计的方式传递。

---

### 单独 vs 完整 Harness 的效果对比

以一个复古游戏制作工具的需求为测试用例：

| 指标 | 单独运行 | 完整 Harness |
|------|----------|--------------|
| 耗时 | 20 分钟 | 6 小时 |
| 成本 | $9 | $200 |
| 质量 | 游戏机制损坏、工作流不直观 | 功能完整的编辑器、可玩关卡、AI 辅助集成 |

单独运行往往产生"看起来令人印象深刻但充满 bug"的应用。Harness 系统性地捕获了：实体连线断裂、填充工具失效、API 路由错误等关键问题。

**重要：22 倍的成本增长是有条件合理的。** 文章明确指出："评估者的核心价值取决于任务复杂度与模型能力的相对关系。"Harness 的每个组件都有成本，只有在任务复杂度超过模型单独能力时才值得投入。

---

### 随模型能力提升持续精简 Harness

当 Claude Opus 4.6 发布后，团队系统性地剥除了不再必要的脚手架：

- Sprint 分解变为可选；模型已能处理更长的连贯会话
- 单次评估移至构建结束，减少中间开销
- 一个数字音频工作站项目在 ~4 小时（$125）内完成

核心洞见：**更强的基础模型不是让 harness 消亡，而是让设计空间转移**——精简低价值脚手架，腾出资源去攻克之前无法企及的难题。

## 二、指导 Bourbon 开发的核心 Mindset

---

### Mindset 1：生成与评估必须分离

自我评估是**结构性偏差**，不是偶发错误。一个 agent 生成输出并评估输出，就像让作者自己校对稿件——盲点是内建的。分离生成与评估，才能建立真正的外部反馈回路。

**对 Bourbon 的含义**：当前 `Agent._run_conversation_loop()` 中，agent 在同一个对话流里既生成又（隐式地）自我判断是否完成。对于质量要求较高的任务，应考虑引入独立的评估 agent 或评估步骤。`evals/` 框架已是这个方向的体现——但评估的触发时机和独立性还可以进一步设计。

---

### Mindset 2：将主观性转化为可测量标准

"质量好"无法驱动迭代，"颜色对比度、字体层级、布局一致性各打多少分"才能。**任何看似主观的质量维度，都可以尝试分解为可操作的子指标。** 可量化意味着可迭代，可迭代意味着可改进。

**对 Bourbon 的含义**：`evals/cases/` 中的断言设计是这个原则的直接实践。问题不是"这个断言对不对"，而是"这个断言能否区分勉强及格和真正优秀"。在设计技能（SKILL.md）的评估维度时，同样要问：我能用几个具体维度代替"这个技能好不好"这个模糊问题吗？

---

### Mindset 3：规划强调范围，而非步骤细节

Planner 的职责是**划定边界**，不是预演实现。过度细化的规划反而适得其反——它把 Generator 锁定在预设路径上，消耗了上下文，却失去了适应性。好的规划定义"要做到什么"，而不是"每一步怎么做"。

**对 Bourbon 的含义**：当 Bourbon 执行复杂的多步任务时，在开始实现前应先用几句话明确任务范围和成功标准，而不是立刻跳入工具调用序列。这与 `superpowers:writing-plans` 技能的设计理念一致——计划应当是可以调整的框架，而不是不可偏离的脚本。

---

### Mindset 4：上下文是稀缺资源，警惕"上下文焦虑"

Claude Sonnet 4.5 在上下文接近满时会提前收尾——不是因为任务完成，而是模型感受到压力。这是一个可观察的行为，值得在 harness 设计中主动防范。实验表明，**重置上下文优于原地压缩**：干净的起点比失真的摘要更可靠，即使看起来"浪费"了前面的上下文。

**对 Bourbon 的含义**：`compression.py` 中的 `ContextCompressor` 和 `microcompact()` 做的是原地摘要——这是文章认为次优的策略。对于长任务，应设计明确的"阶段边界"：在每个阶段结束时生成结构化的工件文件，然后允许干净的上下文切换，而不是把所有上下文压在一个越来越长的对话里。

---

### Mindset 5：基于文件的交接是多智能体协作的基础设施

当 agent 之间有角色分工时，接力棒必须以**持久、可审计的方式**传递。文件不依赖共享内存，不依赖对话上下文的连续性，可以跨越上下文重置。这不是实现细节，而是让分布式 agent 协作可靠运作的根基。

**对 Bourbon 的含义**：技能文件（SKILL.md）本身就是一种文件交接机制——skill 的发现、加载、执行都通过文件系统而非内存传递。在设计复杂的多阶段任务时，应主动利用文件作为 agent 之间的接力点，而不是试图在一个长对话里传递所有状态。

---

### Mindset 6：显式合约防止范围蔓延

Generator 和 Evaluator 在 sprint 前协商"成功标准"，是防止"我以为你要的是这个"问题的**契约机制**。没有显式合约，agent 之间的分工只是表面上的分离——底层仍然是各自对任务的不同理解在暗中冲突。

**对 Bourbon 的含义**：在高风险工具调用（`RiskLevel.HIGH`）前，Bourbon 已有用户确认机制。可以将这个思路延伸：在开始一个复杂任务的实现前，先在对话中明确列出"我计划做 X、Y、Z，完成标准是……"，让用户或评估 agent 确认，而不是一路跑到底再发现方向偏了。

---

### Mindset 7：Prompt 的措辞塑造 Agent 的优化方向

评估标准中使用"museum quality"这样的措辞，直接让设计收敛到特定美学。**Prompt 不只是描述任务，它在为 agent 建立内部判断标准。** 措辞的选择是在隐性地告诉 agent：什么是好，什么是不好。

**对 Bourbon 的含义**：技能文件（SKILL.md）的措辞选择不是可以随意的。每一个描述质量的词——"优雅的"、"简洁的"、"健壮的"——都在为 agent 建立一套价值取向。在写 SKILL.md 时，要问：我选择的这些词，会让 agent 朝哪个方向优化？是我真正想要的方向吗？

---

### Mindset 8：Harness 与模型能力共同进化，定期审计负载组件

随着模型能力提升，harness 设计空间不会收缩，而是**转移**——精简低价值的脚手架，腾出资源去攻克之前无法企及的难题。但这要求定期主动问："现在哪些 harness 组件还是真正必需的？"否则，为旧模型设计的补丁会堆积成遗留债务。

成本与复杂度的开销只在**任务复杂度超过模型单独能力**时才值得。不加判断地使用完整 harness，是另一种浪费。

**对 Bourbon 的含义**：随着 claude-sonnet-4-6 和更新模型的出现，定期审视：当前的错误处理逻辑、上下文压缩策略、风险确认机制——哪些是真正必要的，哪些是历史遗留的过度保守？同时，更强的模型也意味着可以尝试更复杂的多 agent 任务，不要只是用新模型跑旧架构。

---

## 三、总结

| 原则 | 一句话 |
|------|--------|
| 生成与评估分离 | 自我评估是结构性盲点，外部反馈才能驱动真正改进 |
| 主观性可量化 | 找到维度，就能迭代 |
| 规划强调范围 | 定义"做到什么"，而非预演"怎么做每一步" |
| 重置优于压缩 | 警惕上下文焦虑；干净的起点比失真的摘要更可靠 |
| 文件是交接基础设施 | 持久、可审计的工件传递让多 agent 协作可靠 |
| 显式合约 | 编码前约定成功标准，而非事后对齐理解 |
| Prompt 是方向盘 | 措辞塑造 agent 的优化方向，选词即定价值观 |
| Harness 与模型共进化 | 定期审计负载组件；更强的模型解锁更难的问题，而非让 harness 消亡 |
